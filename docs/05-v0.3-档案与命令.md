## 2. 产品句（钉死）

用终端交互替代传统布局的个人站。  
文件系统是分类隐喻；**档案是对象，终端是探索器**——多层是设计。

## 3. 本版范围

| 做 | 不做 |
|----|------|
| 充实 VFS 约定与可逛材料 | 复杂 Person 对象模型大重构 |

## 4. 目录约定（少目录、深材料）

当前顶层保持：

```text
/
├── person
├── projects/
├── thoughts/
└── timeline
```

- 新顶层目录**只在有真实内容时**加（如 `notes/`），不预留空架子。  
- 每个顶层目录目标：有可 `open` 的节点，而不是只有列表命令。  
- `open <目录>` / `open .` / `open *`：打开该目录下可阅读节点（见 §5），不是报「是目录」。

---

## 5. `open` 语义（本刀已定）

| 输入 | 行为 |
|------|------|
| `open <slug\|文件路径>` | 单篇 → main（旧 main demote 进 rail） |
| `open a b c` | 多目标：去重后**最后一项 → main**，其余进 rail |
| `open .` / `open *` | 当前目录下可阅读节点；**首项 → main**，其余进 rail |
| `open projects` / `open thoughts` | 同上，对该目录批量打开 |
| `open timeline` / `open person` | 保持：timeline → 阅读面板；person → 终端摘要（暂不进面板） |

硬顶：主槽 1 + rail ≤ 8；超出截断并提示。  
`prefers-reduced-motion`：批量仍瞬时换位，不串场秀。

## 6. 随后命令（未做，排队）

- `find`：路径向检索，结果可再 `open`  
- `status`：计数 / 最近更新（档案感，非监控大屏）  
- help：按「探索 / 阅读」分组；与上表对齐  
- 目录说明文件（如可 open 的 about）：有文再挂进 VFS  

---

## 7. 成功标准（v0.3 末）

- 连续探索约 5 分钟，不会三步走空。  
- `open thoughts` / `open a b` 行为符合预期，rail 不爆。  
- 进场叙事仍可继续延后；不因「命令变多」而提前做引导动画。
